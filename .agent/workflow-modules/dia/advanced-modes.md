# 高度モード

> メタ判定とクロスモデル検証。基本モードより高い認知負荷を要する。
>
> **制約**: 各モードの出力形式に従うこと。確信度レベルを必ず含めること。

---

|:-----|:-----|

## cross-model (Cross-Model Verification + Cortex Auto)

> `/dia.cross-model` — 異なるモデルによる相互検証 (Cortex API 自動統合)
>
> **起源**: BOU #3 "Debate Partner — 対等な批判者"
> **手段**: `mcp_ochema_ask_cortex` で Gemini 3 Pro に同じ問題を投げ、差分を検出

発動: `/dia cross-model` または「Claude に聞く」「別のAI」「セカンドオピニオン」

### 自動セカンドオピニオン (3フェーズ)

| Phase | 名称 | 内容 |
|:------|:-----|:-----|
| **P1** | 問い投射 | Claude の判定/分析結果を要約し、Gemini に同じ問題を独立で問う |
| **P2** | 応答取得 | `mcp_ochema_ask_cortex` で Gemini 3 Pro の回答を取得 |
| **P3** | 差分分析 | Claude vs Gemini の一致点・相違点・盲点を構造化 |

#### P1: 問い投射テンプレート

```
以下の問題について、独立した分析を行ってください。
他のAIの分析結果は見ないでください。

## 問題
{対象の問題/設計/コード}

## 求める分析
1. 主要な論点の特定
2. リスク/問題点の指摘
3. 改善提案
4. 確信度 (0-100%)
```

#### P2: Cortex 呼び出し

```
mcp_ochema_ask_cortex(
  message="{P1 のプロンプト}",
  model="gemini-2.5-pro",   # デフォルト。gemini-3-pro-preview も可
  system_instruction="あなたは独立した技術レビュアーです。他のAIの意見は参照せず、自分の分析を行ってください。"
)
```

#### P3: 差分分析テーブル

| 項目 | Claude | Gemini | 差分 |
|:-----|:-------|:-------|:-----|
| 主要論点 | ... | ... | 一致/相違 |
| リスク指摘 | ... | ... | 一致/相違 |
| 改善提案 | ... | ... | 一致/相違 |
| 確信度 | ...% | ...% | Δ |
| **盲点** | Gemini が指摘し Claude が見落とした点 | Claude が指摘し Gemini が見落とした点 | — |

## deliberative (三視点反復改善)

> `/dia.deliberative` — 三人の専門家視点で反復改善

発動: `/dia deliberative` または「反復改善」「三視点」

プロセス:

1. **ROUND 1 — 三視点生成**: Expert A(ドメイン専門家), B(批評家), C(実装者)
2. **ROUND 2 — 根拠提示**: 各専門家が根拠を明示、相互矛盾を検出
3. **ROUND 3 — 統合改定**: 三視点を統合し改定版を生成
4. **品質ゲート**: 改善が収束したか評価。未収束なら ROUND 1 へ (最大3サイクル)

## interventional (介入不変性テスト)

> `/dia.interventional` — 判定が条件変更に対して堅牢かテストする
>
> **由来**: Pearl do-calculus に触発 (DX-012 §12)。
> 形式的な因果推論ではなく、認知的ヒューリスティックとして設計。
> **核心問い**: 「この判定は、前提条件を変えても維持されるか？」

発動: `/dia interventional` または「条件を変えても」「介入しても」「前提が崩れたら」

### 3ステップ手続き

| Step | 名称 | 内容 |
|:-----|:-----|:-----|
| **I1** | 前提列挙 | 現在の判定が依存している前提条件を3つ以上列挙する |
| **I2** | 介入シミュレーション | 各前提を「もしこれが偽だったら？」と反転させ、判定がどう変わるか検討 |
| **I3** | 不変性判定 | 全前提反転後も判定が維持 → ROBUST / 1つでも崩壊 → FRAGILE (+ 最弱前提を特定) |

### 出力テーブル

| 項目 | 内容 |
|:-----|:-----|
| 対象判定 | {評価対象の判定/結論} |
| 前提 1 | {前提} → 反転結果: {判定への影響} |
| 前提 2 | {前提} → 反転結果: {判定への影響} |
| 前提 3 | {前提} → 反転結果: {判定への影響} |
| **堅牢性** | ROBUST / FRAGILE |
| **最弱前提** | {最も脆弱な前提とその理由} |
| 確信度 | 介入テスト後の確信度 |

> ⚠️ これは Pearl の do-calculus **ではない**。
> do-calculus が形式的に解決する「交絡」「因果vs相関」の問題を、
> 言語レベルで近似的に回避するヒューリスティック。

## retrospective (反実仮想テスト)

> `/dia.retrospective` — 過去の判定・行動を反実仮想的に再検討する
>
> **由来**: Pearl L3 Counterfactual (The Book of Why, 2018) に触発。
> interventional (前方向) と対をなす**後方向**の認知的ヒューリスティック。
> **核心問い**: 「何を変えていれば、結果は異なっていたか？」
>
> **Pearl L3 との関係** (DX-012 §13 参照):
> Pearl L3 は SCM 内で Abduction→Action→Prediction の3ステップを
> 形式的に実行する。本モードはこれを認知レベルで近似する試み。

発動: `/dia retrospective` または「振り返り」「あのとき」「何が違えば」

### 3ステップ手続き (Pearl L3 対応)

| Step | Pearl L3 | /dia.retrospective | 内容 |
|:-----|:---------|:-------------------|:-----|
| **R1** | Abduction | 状況復元 | 過去の判定時点の情報・前提・制約を可能な限り忠実に復元する |
| **R2** | Action | 仮説的変更 | 「もし X を変えていたら」— 1つの変数 (判断基準, 情報, 行動) を仮説的に変更 |
| **R3** | Prediction | 結果予測 | 変更した世界で、結果がどう変わっていたかを推論する |

### 出力テーブル

| 項目 | 内容 |
|:-----|:-----|
| 対象判定 | {過去の判定/行動とその結果} |
| **R1 状況復元** | 当時の情報: {}, 前提: {}, 制約: {} |
| **R2 仮説的変更** | 変更変数: {}, 変更内容: {} |
| **R3 結果予測** | 予測される代替結果: {} |
| **学習** | この反実仮想から得られる教訓は何か？ |
| **モデル更新** | Δ層のどの定理/WF を修正すべきか？ |
| 確信度 | 反実仮想の信頼度 |

### interventional との対比

| | interventional | retrospective |
|:---|:---|:---|
| **方向** | 前方向 (これから) | 後方向 (過去) |
| **問い** | 「条件を変えても判定は維持されるか？」 | 「何を変えれば結果は変わったか？」 |
| **Pearl** | L2 (Intervention) | L3 (Counterfactual) |
| **目的** | 判定の堅牢性テスト | 失敗からの学習 |

> ⚠️ これは Pearl L3 の形式的実装**ではない**。
> SCM が必要とする背景変数の完全な特定は LLM には不可能。
> ただし「振り返って学ぶ」認知プロセスとして、
> React→Plan→**Reflect**ループの Reflect 部分を強化する。
