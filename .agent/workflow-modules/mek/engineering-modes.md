# エンジニアリングモード (Engineering Modes)

> S2 Mekhanē — 品質向上と簡素化のための5つの工学的手法。

## Constraints

- 出力はテーブル形式で構造化する
- 定量的な指標（削減率、使用頻度等）を含める
- 改善前後の差分を明示する

---

## yagni — 不要機能排除

**CCL**: `/mek.yagni` = `/mek-{filter=unnecessary}`
**目的**: 不要な機能を排除する
**発動**: `/mek yagni` または「YAGNI」「いらない」「過剰」

### Process

1. 現在の機能一覧を確認
2. 各機能の使用頻度を評価
3. 使われていない機能を特定
4. 削除の影響を評価
5. 削除候補 + 理由を出力

### Output Format

**[S2 Mekhanē: YAGNI]**

| フィールド | 内容 |
|:-----------|:-----|
| 対象 | {評価対象} |
| 削除候補 | {機能1}: 使用頻度 0%, {機能2}: 「いつか使うかも」 |
| 保持 | {機能3}: アクティブに使用中 |
| 削減効果 | 複雑度 -{N}% |

---

## dry — 重複排除

**CCL**: `/mek.dry` = `/mek-{filter=duplicate}`
**目的**: 重複コード・ロジックを排除
**発動**: `/mek dry` または「DRY」「重複」「共通化」

### Process

1. 重複箇所を検出
2. 共通パターンを抽出
3. 抽象化の方法を決定
4. リファクタリング実施
5. 共通化後 + 削減効果を出力

### Output Format

**[S2 Mekhanē: DRY]**

| フィールド | 内容 |
|:-----------|:-----|
| 重複検出 | {ファイル1}:{行} ≈ {ファイル2}:{行} |
| 共通化案 | {抽出した共通関数/モジュール} |
| Before | {重複行数}行 |
| After | {共通化後行数}行 |
| 削減 | {N}% (LOC) |

---

## observability — 可観測性

**CCL**: `/mek.observability` = `/mek+{feature=monitoring}`
**目的**: システムの可観測性を設計する
**発動**: `/mek observability` または「可観測性」「ログ」「メトリクス」「トレース」

### Process

1. 観測対象を特定
2. ログ設計（何を記録するか）
3. メトリクス設計（何を計測するか）
4. トレース設計（どう追跡するか）
5. 可観測性設計を出力

### Output Format

**[S2 Mekhanē: Observability]**

| レイヤー | 項目 | 詳細 |
|:---------|:-----|:-----|
| Logs | {イベント1} | レベル: {INFO/WARN/ERROR} |
| Metrics | {メトリクス1} | 単位: {count/ms/bytes} |
| Traces | {スパン1} → {スパン2} | {コンテキスト} |

---

## substitute — 代替手段

**CCL**: `/mek.substitute` = `/mek+{strategy=alternative}`
**目的**: 既存の方法の代替手段を発見する
**発動**: `/mek substitute` または「代替」「別の方法」「置き換え」

### Process

1. 現在の方法を特定
2. 代替候補を列挙
3. トレードオフを比較
4. 代替の適合度を評価
5. 代替候補 + 比較を出力

### Output Format

**[S2 Mekhanē: Substitute]**

| 方法 | 長所 | 短所 | 推奨 |
|:-----|:-----|:-----|:----:|
| 現行: {現在の方法} | {長所} | {短所} | — |
| 代替1: {候補1} | {長所} | {短所} | {○/×} |
| 代替2: {候補2} | {長所} | {短所} | {○/×} |

---

## nudge — 行動誘導

**CCL**: `/mek.nudge` = `/mek+{focus=behavior}`
**目的**: ユーザー行動を誘導する設計を行う
**発動**: `/mek nudge` または「ナッジ」「行動誘導」「デフォルト設計」

### Process

1. 望ましい行動を定義
2. 現在のバリアを分析
3. ナッジ手法を選択
4. 倫理的考慮を確認
5. ナッジ設計を出力

### Output Format

**[S2 Mekhanē: Nudge]**

| フィールド | 内容 |
|:-----------|:-----|
| 望ましい行動 | {誘導したい行動} |
| バリア | {障壁1}, {障壁2} |
| デフォルト | {デフォルト値の設計} |
| フレーミング | {情報の見せ方} |
| 社会的証明 | {他者の行動を示す} |
| 倫理確認 | 操作的でないことを確認済み |

---

## Reminder

- 出力はテーブル形式。装飾より情報構造
- 定量的指標を必ず含める
