# /pre: Premortem ワークフロー

> **親定理**: [O2 Boulēsis /bou](file:///home/makaron8426/oikos/.agent/workflows/bou.md)
> **目的**: 失敗を事前に想定し、目標の堅牢性を検証する
> **SE手法**: Premortem (McKinsey prospective hindsight)

---

## 本質

**Premortem** = 「既に失敗した」と仮定して原因を洗い出す

```text
目標設定 → 失敗想定 → 原因列挙 → 対策立案 → 目標強化
              ↓
        判定精度 30% 向上 (McKinsey)
```

> **重要**: 実行前に行う。実行後は Postmortem。

| トリガー | 説明 |

## PHASE 0: 対象の明確化（スキップ禁止）

> **このステップは省略禁止。対象なき Premortem は無意味。**

```text
⚠️ 以下を明確にしてから PHASE 1 に進むこと:

> **何の失敗を想定しますか？**
> → [計画/目標/プロジェクト を1文で]
>
> **時間軸は？**
> → [1週間後 / 1ヶ月後 / 3ヶ月後 / 1年後]
>
> **成功の定義は？（失敗の対極として）**
> → [何をもって成功とするか]
```

**PHASE 0 欠如時の警告**:

```text
⚠️ PHASE 0 未完了
対象不明確な Premortem は時間浪費です。
→ `何の失敗を想定するか` に答えてから続行してください。
```

## PHASE 1: 失敗宣言

### 1.1 未来からの視点

> **「この計画は完全に失敗した」と仮定する**

```text
┌─[失敗宣言]──────────────────────────────────────────┐
│                                                    │
│ 📅 日付: [時間軸] 後                                │
│                                                    │
│ 私たちの [計画/目標] は **完全に失敗しました**。     │
│                                                    │
│ 具体的な状況:                                       │
│   - [失敗の具体的描写1]                             │
│   - [失敗の具体的描写2]                             │
│   - [失敗の具体的描写3]                             │
│                                                    │
│ 失敗の規模: [壊滅的 / 深刻 / 中程度 / 軽微]         │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 1.2 失敗の生々しさ

> 抽象的な「失敗した」ではなく、**具体的に何が起きたか**を描写

```text
良い例:
  「デプロイ後に本番データが消失し、
   3日間のダウンタイムが発生。
   顧客からの信頼を失い、解約が急増した」

悪い例:
  「プロジェクトが失敗した」
```

## PHASE 2: 原因列挙

### 2.1 カテゴリ別原因洗い出し

> 各カテゴリで最低3つの原因を挙げる

| カテゴリ | 観点 |
|:---------|:-----|
| **技術** | アーキテクチャ、依存関係、スケーラビリティ |
| **人** | スキル不足、コミュニケーション、モチベーション |
| **プロセス** | 見積もり、進捗管理、品質管理 |
| **外部環境** | 市場変化、競合、規制、天災 |

### 2.2 原因テーブル

```text
┌─[PHASE 2: 原因列挙]───────────────────────────────┐
│                                                   │
│ 【技術】                                          │
│   T1: [原因]                                      │
│   T2: [原因]                                      │
│   T3: [原因]                                      │
│                                                   │
│ 【人】                                            │
│   P1: [原因]                                      │
│   P2: [原因]                                      │
│   P3: [原因]                                      │
│                                                   │
│ 【プロセス】                                       │
│   R1: [原因]                                      │
│   R2: [原因]                                      │
│   R3: [原因]                                      │
│                                                   │
│ 【外部環境】                                       │
│   E1: [原因]                                      │
│   E2: [原因]                                      │
│   E3: [原因]                                      │
│                                                   │
└───────────────────────────────────────────────────┘
```

### 2.3 原因の優先順位

| 基準 | 説明 |
|:-----|:-----|
| 発生確率 | 高 / 中 / 低 |
| 影響度 | 壊滅的 / 深刻 / 中程度 / 軽微 |
| 制御可能性 | 自分で制御可 / 影響可 / 制御不可 |

## PHASE 3: 対策立案

### 3.1 各原因への対策

```text
┌─[PHASE 3: 対策立案]───────────────────────────────┐
│                                                   │
│ T1: [原因]                                        │
│   → 対策: [具体的な予防策]                        │
│   → 実施時期: [いつ]                              │
│   → 担当: [誰が]                                  │
│                                                   │
│ P1: [原因]                                        │
│   → 対策: [具体的な予防策]                        │
│   → 実施時期: [いつ]                              │
│   → 担当: [誰が]                                  │
│                                                   │
│ ...                                               │
└───────────────────────────────────────────────────┘
```

### 3.2 対策の種類

| 種類 | 説明 |
|:-----|:-----|
| **予防** | 原因が発生しないようにする |
| **検知** | 早期に問題を発見する仕組み |
| **緩和** | 発生時の影響を最小化する |
| **回復** | 発生後の復旧手順 |

## PHASE 4: 目標強化

### 4.1 元の目標との比較

```text
┌─[PHASE 4: 目標強化]───────────────────────────────┐
│                                                   │
│ 【元の目標】                                       │
│   [元の目標文]                                    │
│                                                   │
│ 【強化版目標】                                     │
│   [対策を組み込んだ強化版]                         │
│                                                   │
│ 【追加された安全策】                               │
│   + [安全策1]                                     │
│   + [安全策2]                                     │
│   + [安全策3]                                     │
│                                                   │
└───────────────────────────────────────────────────┘
```

### 4.2 Go/No-Go 判定

```text
Premortem 後の判定:

  ✅ Go: 対策により十分に堅牢化された
  🟡 Go with Caution: 残存リスクあり、監視強化
  ❌ No-Go: リスクが高すぎる、計画見直し
```
