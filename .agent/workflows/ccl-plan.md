# /ccl-plan: 計画策定マクロ

> **CCL**: `@plan = /bou+_/chr_/hod_/s+~(/p*/k)_V:{/dia}_/pis_/dox-`
> **用途**: 何かを始める前に計画を練りたいとき
> **認知骨格**: Prior → Likelihood → Posterior
> **v2.0**: 各ステップの定理マッピングを明示化 (24定理活用深化 Phase 2)

## 展開 — 定理マッピング付き

| 相 | ステップ | 定理 | 意味 |
|:---|:---------|:-----|:-----|
| Prior | `/bou+` | **O2 Boulēsis** | 意志を詳細化: 何を達成したいか |
| Prior | `/chr` | **K2 Chronos** | 時間資源確認: いつまでに、どれくらいかかるか |
| Prior | `/hod` | **P2 Hodos** | 経路設計: どの道を行くか、代替経路はあるか |
| Likelihood | `/s+` | **S1-S4 全体** | 戦略を全面展開: 測定(S1)→方法(S2)→品質基準(S3)→実践(S4) |
| Likelihood | `~(/p*/k)` | **P1-P4 × K1-K4** | 環境と文脈を振動させて練る |
| Likelihood | `V:{/dia}` | **A2 Krisis** | 判定で検証ゲート: 計画は妥当か |
| Posterior | `/pis` | **H2 Pistis** | 計画の確信度を測定 |
| Posterior | `/dox-` | **H4 Doxa** | 計画を軽量記録 |

| **P1 Khōra** | 作業空間のスコープは明確か？ |
| **P2 Hodos** | 経路に代替案はあるか？ |
| **P3 Trokhia** | 既知のパターン/アンチパターンはあるか？ |
| **P4 Tekhnē** | 技術選定は適切か？ |
| **K1 Eukairia** | 今が好機か？ |

## 出力品質ゲート (旧@ground 組み込み)

> **由来**: @ground (`/tak-*/bou+{6w3h}~/p-_/ene-`) を instruction レベルで統合。
> **発動**: `/pis` (確信度測定) の前に、以下が埋まっているか自問する。

| # | 項目 | 問い |
|:--|:-----|:-----|
| W1 | **Who** | 誰が実行するか？ |
| W2 | **What** | 何を作る/変えるか？ |
| W3 | **Why** | なぜ必要か？ 代替案を棄却した理由は？ |
| W4 | **Where** | どこに配置/影響するか？ |
| W5 | **When** | いつまでに？ 依存関係は？ |
| H1 | **How** | どう実装するか？ |
| H2 | **How much** | スコープの大きさ — 見積もりは？ |
| H3 | **How to verify** | どう検証するか？ |

> 全項目を埋める必要はない。**埋まらない項目がある = 計画が不完全** のシグナル。

## Context Engineering 原則 (2026-02-15 追加)

> **由来**: Anthropic + Karpathy + Lütke の独立定義。HGK 第零原則「意志より環境」の外部裏付け。
> **発動**: 計画策定時に常に意識する3原則。

| # | 原則 | 問い |
|:--|:-----|:-----|
| CE1 | **タスク分解 > プロンプト最適化** | 単一プロンプトで解決しようとしていないか？ タスクを分解し、WF/CCL として構造化できないか？ |
| CE2 | **情報注入の設計** | コンテキストに何を含めるか明示したか？ 必要な情報はツールで取得するか、事前に注入するか？ |
| CE3 | **出力ルーティング** | 結果をどこに流すか？ ファイル保存、次の WF への入力、ユーザーへの提示のどれか？ |
