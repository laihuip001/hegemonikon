# /chk ワークフロー

> **Hegemonikón Module**: M7 Dokimē (検証)
> **目的**: 仮説、計画、コード、または成果物に対して、客観的な検証と批評を行う。

---

## 発動条件

- `/chk` コマンド実行時
- ユーザーから「テストして」「批評して」「レビューして」と求められた時
- `/plan` や `/do` の成果物に対する品質チェック時

---

## 実行モード

| モード | コマンド | 説明 |
|--------|----------|------|
| **Validate** | `/chk` | 一般的な検証・テスト設計・事実確認 |
| **Synedrion** | `/chk synedrion` | **偉人評議会**による多角的批評 |

---

## Mode 1: Validate (通常検証)

### Process
1. **対象定義**: 何を検証するか（仮説、コード、設計）
2. **検証法選定**: 直接検証、反証テスト、A/Bテスト、プレモーテム
3. **実行**: 検証を実行、または検証計画を提示
4. **判定**: 合格/不合格/要修正

### Output Format
```markdown
## ✅ Verification Report

- **対象**: [検証対象]
- **結果**: [Pass / Fail / Warn]

### 検証項目
- [x] 条件A: [結果]
- [ ] 条件B: [失敗/未検証] - [理由]

### 推奨アクション
- [アクション1]
```

## Mode 2: Synedrion (偉人評議会)

6人の異なる哲学的視点を持つ評議員を召喚し、成果物を多角的に批評する。

### 評議員 (The Six Critics)
1. 🔓 **チューリング** (Logic): 論理性、計算可能性
2. ⚡ **フォード** (Efficiency): 効率、無駄の排除
3. 🧪 **ファインマン** (Clarity): 明確さ、説明可能性
4. ⚔️ **ベゾス** (Customer): 顧客視点、長期的価値
5. 🏛️ **ダ・ヴィンチ** (Structure): 構造美、シンプルさ
6. 🎨 **ジョブズ** (Experience): UX、感性、完成度

### Process
1. **招集**: 対象物に対し、全評議員が独立して評価。
2. **異議申し立て**: 「承認(✅)」以外の評価を持つ評議員のみが発言。
3. **統合**: 全会一致なら即承認。異議があれば修正案を提示。

### Output Format

```markdown
┌─[Hegemonikón]──────────────────────┐
│ M7 Dokimē: Synedrion Protocol      │
│ 招集: 6名の評議員                  │
│ 判定: [全会一致 / 異議あり]        │
└────────────────────────────────────┘

## 🏛️ Synedrion Review

| 評議員 | 視点 | 評価 | コメント |
|--------|------|------|----------|
| ⚡ フォード | 効率 | ⚠️ | 工数が多すぎる。ステップを減らせないか？ |
| 🎨 ジョブズ | 体験 | 🔴 | 美しくない。ユーザーはこの画面で迷うはずだ。 |
| (他は承認のため省略) | | | |

### Synthesized Advice (統合助言)
効率とUXに課題あり。特に[指摘箇所]の簡略化を推奨する。
```

## Cognitive Armory (思考ツール)

必要に応じて以下のツールを使用する：

- **Devil's Advocate**: あえて反対の立場をとる
- **Pre-Mortem**: 「プロジェクトが失敗した」と仮定して原因を探る
- **10th Man Rule**: 全員賛成なら、一人は義務として反論する
