# /poc: Timeboxed 探索ワークフロー

> **親定理**: [O3 Zētēsis /zet](file:///home/laihuip001/oikos/.agent/workflows/zet.md)
> **目的**: 不確実性を実験で下げる — 仮説検証型探索
> **SE手法**: Spike / Proof of Concept

---

## 本質

**Spike/PoC** = 「やってみないとわからない」を解決する

```text
不確実性 → Timebox 設定 → 最小実装 → 学び
                ↓
          不確実性が下がる
```

> **重要**: PoC のコードは捨てる前提。品質より速度。

---

## 発動条件

| トリガー | 説明 |
|:---------|:-----|
| `/poc` | 探索開始 |
| `/poc [仮説]` | 特定の仮説を検証 |
| `/poc 30m` | 30分 Timebox で探索 |

---

## PHASE 0: 探索目的の明確化（スキップ禁止）

> **このステップは省略禁止。目的なき探索は迷走。**

```text
⚠️ 以下を明確にしてから PHASE 1 に進むこと:

> **何を検証しますか？**
> → [検証対象を1文で]
>
> **成功の定義は？**
> → [「これができたら成功」を具体的に]
>
> **Timebox は？**
> → [30分 / 1時間 / 2時間 / 半日]
>
> **失敗しても学べることは？**
> → [失敗から得られる情報]
```

**PHASE 0 欠如時の警告**:

```text
⚠️ PHASE 0 未完了
目的なき探索は時間浪費です。
→ `何を検証する？成功の定義は？` に答えてから続行してください。
```

## PHASE 1: 仮説設定

### 1.1 仮説の言語化

```text
「〇〇すれば△△になるはず」形式で記述:

仮説: ___________________________________________________________

検証方法: _______________________________________________________

予想される結果: _________________________________________________
```

### 1.2 仮説の質チェック

| チェック項目 | 確認 |
|:-------------|:-----|
| 検証可能か？ | [ ] |
| Timebox 内に完了するか？ | [ ] |
| 結果が明確に判断できるか？ | [ ] |

---

## PHASE 2: 最小実装

### 2.1 実装ルール

> **品質より速度。捨てる前提。**

```text
✅ やること:
   - 仮説検証に必要な最小限のコード
   - ハードコードOK
   - テストなしOK
   - コメントなしOK

❌ やらないこと:
   - エラーハンドリングの完備
   - リファクタリング
   - ドキュメント整備
   - 他機能との統合
```

### 2.2 実装開始

```text
┌─[Hegemonikón]──────────────────────────────────────────┐
│ ⏱️ Timebox: [設定時間]                                  │
│ 🎯 仮説: [検証対象]                                     │
│                                                        │
│ → 実装開始                                              │
└────────────────────────────────────────────────────────┘
```

### 2.3 中間チェック (半分経過時)

```text
Timebox 50% 経過:
  - 順調に進んでいるか？
  - 方向転換が必要か？
  - 仮説を修正すべきか？
```

### 3.2 学びの抽出

```text
学んだこと:
  1. [学び1]
  2. [学び2]
  3. [学び3]

予想外だったこと:
  - [予想外1]
```

### 3.3 次のアクション

| 結果 | 次のアクション |
|:-----|:---------------|
| ✅ 成功 | → 本実装へ (`/s` で設計) |
| ❌ 失敗 | → 別仮説で再 PoC or 撤退 |
| 🟡 部分的 | → 追加 PoC で深掘り |

## ベストプラクティス

### Timebox の目安

| 不確実性レベル | 推奨 Timebox |
|:---------------|:-------------|
| 低（やったことある） | 30分 |
| 中（似たことはある） | 1-2時間 |
| 高（全く未知） | 半日 |

### PoC の連鎖

```text
PoC 1: 基本動作確認 (30m)
  ↓
PoC 2: 性能検証 (1h)
  ↓
PoC 3: 統合テスト (2h)
  ↓
本実装 (/s)
```

```text
→ /zet (X-OO): 再探求 — 新たな問いが見つかった場合

```
