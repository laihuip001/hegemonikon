# /why: 根本原因分析ワークフロー (Five Whys)

> **親定理**: [O3 Zētēsis /zet](file:///home/laihuip001/oikos/.agent/workflows/zet.md)
> **目的**: 根本原因を問い続けて表層を超える
> **SE手法**: Five Whys (トヨタ生産方式)

---

## 本質

**Five Whys** = 「なぜ？」を繰り返して根本原因に到達する

```text
問題発生 → Why 1 → Why 2 → Why 3 → Why 4 → Why 5
   ↓         ↓        ↓        ↓        ↓        ↓
 表層     直接原因  間接原因 プロセス システム  根本原因
```

> **重要**: 5回は目安。3回で到達することも、7回必要なこともある。

---

## 発動条件

| トリガー | 説明 |
|:---------|:-----|
| `/why` | 現在の問題に対して Five Whys |
| `/why [問題]` | 特定の問題に対して Five Whys |
| `/why deep` | 7回まで深掘り |

---

## PHASE 0: 問題の明確化（スキップ禁止）

> **このステップは省略禁止。問題不明確な分析は無意味。**

```text
⚠️ 以下を明確にしてから PHASE 1 に進むこと:

> **何が問題ですか？**
> → [問題を1文で具体的に]
>
> **いつ発生しましたか？**
> → [初回発生 / 再発 / 継続中]
>
> **影響範囲は？**
> → [局所 / システム全体 / ユーザー影響あり]
>
> **期待される状態は？**
> → [問題がない場合の正常な状態]
```

**PHASE 0 欠如時の警告**:

```text
⚠️ PHASE 0 未完了
問題不明確な Five Whys は方向を見失います。
→ `何が問題か` に答えてから続行してください。
```

## PHASE 1: Five Whys 実行

### 1.1 Why の連鎖

```text
┌─[PHASE 1: Five Whys]───────────────────────────────┐
│                                                    │
│ 問題: [PHASE 0 で定義した問題]                      │
│                                                    │
│ Why 1: なぜこの問題が起きたのか？                   │
│   → [直接原因]                                     │
│                                                    │
│ Why 2: なぜ [Why 1 の回答] が起きたのか？           │
│   → [間接原因]                                     │
│                                                    │
│ Why 3: なぜ [Why 2 の回答] が起きたのか？           │
│   → [プロセス原因]                                 │
│                                                    │
│ Why 4: なぜ [Why 3 の回答] が起きたのか？           │
│   → [システム原因]                                 │
│                                                    │
│ Why 5: なぜ [Why 4 の回答] が起きたのか？           │
│   → [根本原因 ← ここが本当の原因]                  │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 1.2 各 Why の深さの目安

| Why | 深さ | 観点 |
|:----|:-----|:-----|
| **Why 1** | 直接原因 | 何が直接引き金になったか |
| **Why 2** | 間接原因 | その引き金を許容した条件は |
| **Why 3** | プロセス原因 | どのプロセスが失敗したか |
| **Why 4** | システム原因 | どのシステム設計が問題か |
| **Why 5** | 根本原因 | 何を変えれば再発しないか |

### 1.3 分岐がある場合

> 1つの Why に複数の回答がある場合は分岐する

```text
Why 2: なぜ [Why 1] が起きたのか？
  → 2a: [原因A]
  → 2b: [原因B]

→ 両方を追跡するか、影響度が高い方を選択
```

## PHASE 2: 根本原因の検証

### 2.1 検証質問

```text
┌─[PHASE 2: 根本原因検証]────────────────────────────┐
│                                                    │
│ 候補となる根本原因: [Why 5 の回答]                  │
│                                                    │
│ 検証質問:                                          │
│   Q1: この原因を解決したら、問題は再発しないか？    │
│       → [Yes/No] [理由]                            │
│                                                    │
│   Q2: この原因は制御可能か？                        │
│       → [制御可 / 影響可 / 制御不可]               │
│                                                    │
│   Q3: Why をもう一段深掘りできるか？                │
│       → [はい → Why 6 へ / いいえ → 根本原因確定]  │
│                                                    │
│ 判定: [根本原因確定 / さらに深掘り必要]            │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 2.2 根本原因の特徴

| 特徴 | 説明 |
|:-----|:-----|
| **制御可能** | 自分たちで変更できる |
| **再発防止** | 解決すれば問題が繰り返されない |
| **説明力** | なぜ問題が起きたかを論理的に説明できる |

---

## PHASE 3: 対策立案

### 3.1 対策の層

```text
┌─[PHASE 3: 対策立案]────────────────────────────────┐
│                                                    │
│ 【根本対策】← 最優先                               │
│   対象: [Why 5 の根本原因]                         │
│   対策: [根本原因を解消するアクション]              │
│   効果: [再発防止]                                 │
│                                                    │
│ 【プロセス対策】                                   │
│   対象: [Why 3 のプロセス原因]                     │
│   対策: [プロセス改善アクション]                    │
│   効果: [早期検知]                                 │
│                                                    │
│ 【応急対策】                                       │
│   対象: [Why 1 の直接原因]                         │
│   対策: [今すぐの緩和アクション]                    │
│   効果: [影響最小化]                               │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 3.2 対策の優先順位

| 優先度 | 対象 | アクション |
|:-------|:-----|:-----------|
| **1** | 根本原因 | 再発防止策を実装 |
| **2** | プロセス | 検知・監視を強化 |
| **3** | 直接原因 | 応急処置で影響を抑える |

## ベストプラクティス

### よくある落とし穴

| 落とし穴 | 対策 |
|:---------|:-----|
| **5回で止まる** | 必要なら6回、7回続ける |
| **1つの線しか追わない** | 分岐がある場合は両方追跡 |
| **「人のせい」で止まる** | 「なぜその人がそう行動したか」を問う |
| **抽象的すぎる回答** | 具体的な事実・行動を記述 |

### 「人のせい」回避

```text
❌ 悪い例:
  Why 3: なぜテストが漏れたか？
    → 「担当者が怠慢だったから」← 責任追及で終わる

✅ 良い例:
  Why 3: なぜテストが漏れたか？
    → 「テストケース一覧がなく、何をテストすべきか不明確だった」
  Why 4: なぜテストケース一覧がなかったか？
    → 「テスト計画のプロセスが定義されていなかった」← システム原因
```
