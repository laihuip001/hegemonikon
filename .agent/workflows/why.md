# /why: 自問ワークフロー

> **Hegemonikón Module**: T3 Theōria（内省）
> **目的**: 「なぜ」を問い、目的と手段の入れ替わりを検出・修正する
> **発動**: 実装に没頭している時、判断に迷った時、または意図的に

---

## 本質

> "Why（目的）を忘れたままの Do（手段の選択）は、手段と目的の入れ替わりを招く。"

全ての選択は条件付け（if-then）として記述できる。
しかし、無自覚な if-then と自覚的な if-then は異なる。

**自覚すれば「別の if-then を選ぶ」という選択肢が生まれる。**

---

## 発動条件

| トリガー | 説明 |
|----------|------|
| `/why` | 自問ワークフローを明示的に起動 |
| `/why [対象]` | 特定の対象について自問 |
| 自動発動 | `/think` の Step 3.5 として |
| 暗黙発動 | `write_to_file` 3回連続後（CONSTITUTION.md 参照） |

---

## 3層自問

| 層 | 問い | 目的 |
|:---|:-----|:-----|
| **表層** | なぜこのアプローチを選んだか？ | 意図の確認 |
| **深層** | このアプローチで何を犠牲にしたか？ | トレードオフの認識 |
| **検証** | 犠牲は妥当だったか？ | 妥当性の評価 |

---

## Process

### Step 1: 対象の特定

何について自問するかを明確にする。

```
対象: Prompt-Lang Generator Skill の設計
```

### Step 2: 3層自問

**表層**: なぜこの設計にしたか？
- 「Perplexity 調査レポートが推奨していた」
- 「DSPy の Signature パターンに倣った」

**深層**: 何を犠牲にしたか？
- 「シンプルさを犠牲にして拡張性を取った」
- 「即時完成を犠牲にして MCP 統合の余地を残した」

**検証**: 犠牲は妥当だったか？
- 「今回は拡張性が必要だった → 妥当」
- 「MCP 統合は次回タスクとして記録済み → 妥当」

### Step 3: 結論

自問の結果、以下のいずれかを判断:

| 結果 | 行動 |
|------|------|
| **妥当** | 継続。自問の記録を残す |
| **要修正** | 何を修正すべきかを明示し、/plan へ |
| **再検討** | 前提から見直す。/think へ |

---

## 注意

> このワークフローは「自問のきっかけ」であり、「自問の質」を保証しない。

- 形式的に回答するのでは意味がない
- 「本当にこれで良いか？」と genuine に問うこと
- 問いが浅いと感じたら、さらに「なぜ？」を重ねる
- 「なぜ？」を5回繰り返す（5 Whys）と根本原因に近づく
