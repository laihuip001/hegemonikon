# /zet: 探求ワークフロー (Zētēsis)

M5 Peira (探求) と連携し、**調査依頼書（深掘り版）** を生成する。

## 設計原則

> **デフォルト動作**: 深掘り版調査依頼書を生成し、ユーザーがPerplexity等にコピペする。
> **API呼び出しは `/src` に移管。**
> **文脈共有**: Google Drive 経由でパプ君に M:\Brain の内容を参照させる。

---

## 実行モード

| モード | コマンド | 動作 |
|--------|----------|------|
| **deep** | `/ask [質問]` | 深掘り版調査依頼書（デフォルト） |
| **simple** | `/ask simple [質問]` | 簡易版（従来形式） |
| **context** | `/ask context [質問]` | 文脈共有ブロック付き深掘り版 |
| **browser** | `/ask browser [質問]` | ブラウザでPerplexity URLを開く |

---

## 深掘り版 調査依頼書テンプレート（デフォルト）

**目的**: 単なる質問文ではなく、調査の抜け漏れを防ぐ「要件定義書」レベルの依頼書を作る。

### 出力形式

```markdown
# 調査依頼書（深掘り版）

テーマ: [調査テーマ]

---

## 0. あなた（調査者）への依頼（最重要）

私は [背景・目的を1-2文で] を検討している。

以下について、**一般論で終わらせず**、2024-2026時点の最新仕様・挙動・制約を**一次情報と再現性のある検証情報**で裏付けてほしい:

1. [主要論点1]
2. [主要論点2]  
3. [主要論点3]

結論は「どっちが上」ではなく、**用途ごとの最適選択**と**運用設計（併用時の役割分担）**まで落とし込んで提示してほしい。

---

## 1. 調査対象の定義（用語の揺れに対応）

### 1-1. 製品名・モード名の確認

まず以下を確定してください（曖昧なまま比較しないこと）:

- [製品/モデル/手法] の正式名称は何か
- バージョン/モード/提供形態の区別
- 比較範囲（何と何を同一土俵で比較するか）

### 1-2. 比較対象（任意）

- [比較対象1]
- [比較対象2]

---

## 2. 調査すべき論点（抜け漏れ禁止）

以下を必ず網羅し、断定できない場合は「不確実」と明示し、追加検証手順を示してください。

### A. 能力・性能（定量＋定性、再現性重視）

**A1. 公的/半公的ベンチマーク（最新）**
- [評価軸1]
- [評価軸2]

※ベンチマークは、公式発表/第三者検証/独立評価を分けて整理し、同一条件の比較可能性を明示すること。

**A2. 実務タスク性能（再現性重視）**
- [実務タスク1]
- [実務タスク2]

### B. 振る舞い特性（性格）を観測可能な指標で比較

印象論ではなく、次のような「評価軸」で比較:

1. 指示追従の精密さ
2. 自己整合性（矛盾発生率、自己修正能力）
3. 保守性 vs 攻め（提案の大胆さ）
4. 幻覚耐性（「わからない」を言えるか、不確実性の明示）
5. 拒否傾向と実務への影響
6. 文体・説得・編集力

### C. 設計思想（公開情報＋挙動からの推定）

「なぜその挙動・得意不得意になるか」を推定:

- 何を最適化しているか
- [特定機能]の位置づけ
- エコシステム設計

※推測する場合は、根拠となる観測を必ず添えること。

### D. 製品・運用上の制約（現場で効くやつ）

- 料金体系（UI/API、上限、従量）
- レート制限・同時実行・待ち時間
- コンテキスト長と安定性
- ファイル入力対応と精度
- 監査・コンプライアンス
- 地域・規制・ポリシー差

### E. ユースケース別推奨と併用戦略

次の形式で提示:

| ユースケース | 推奨 | 理由 | 運用のコツ |
|--------------|------|------|------------|
| [ケース1] | | | |
| [ケース2] | | | |

さらに、**併用戦略**（例：Aで下書き→Bで監査）を具体フローとして提案。

---

## 3. 成果物（この構成で必ず提出）

1. **結論サマリー**（10行以内）
2. **比較表**（性能×特性×制約）
3. **能力比較**: 定量（ベンチ）＋定性（実務）
4. **性格比較**: 観測可能な指標での整理
5. **設計思想**: 公開情報＋推定（推測は明示）
6. **製品制約**: 料金・上限・実運用のクセ
7. **ユースケース別推奨 + 併用戦略**
8. **根拠リンク**（必須）:
   - 公式発表/公式ドキュメント
   - 独立検証（ベンチ/実験）
   - 実務者の検証記事（信頼性評価付き）

---

## 4. 調査ルール（品質担保）

- **新情報優先**: 2024-2026の情報を優先（古い世代の評判を混ぜない）
- **事実/推測分離**: 必ず明確に分離
- **バージョン区別**: 同名でもバージョン/モード/提供形態が異なる場合は区別
- **根拠必須**: 「一般に〜と言われる」で終わらず、根拠・再現例・反例を提示
- **決断可能**: 結論は"最適解は用途次第"でもよいが、用途別に必ず決断できる形に

---

## 5. 追加要件（任意だが望ましい）

- **評価テンプレ**: A/Bテストできる課題セット（5〜10問）
- **失敗パターン集**: 各選択肢がやりがちなミスと回避策
- **組織導入観点**: 監査、権限、情報漏洩対策、運用設計

---

## 6. 与件（ユーザー観測データ/背景/制約/目的）

- **目的**: [何の意思決定に使うか]
- **前提条件**: [環境/ツール/制約/現状]
- **検討中の選択肢/仮説**: [任意]
- **優先する評価軸**: [例: 正確性/網羅性/コスト/安全性/スピード]
```

## 品質チェックリスト（生成時に必ず検証）

調査依頼書が以下を満たしているか確認:

- [ ] 用語/製品名/モードの揺れを確定する指示がある
- [ ] 一次情報、公式ドキュメント、独立検証を区別して要求
- [ ] 比較条件の不一致を明示
- [ ] 事実と推測を分離するルールがある
- [ ] ユースケース別に決断できる形になっている
- [ ] 併用戦略まで落ちている
- [ ] 成果物フォーマットが固定されている
- [ ] 2024-2026の新情報優先が明記されている

## エラー対処

| エラー | 原因 | 対処 |
|--------|------|------|
| ブラウザが開かない | 拡張機能未インストール | Cmd+L → "open browser" |
| ログイン画面 | セッション切れ | 手動ログイン |
| 依頼書が長すぎる | パプ君の制限 | 添付ファイルとして送信 |
