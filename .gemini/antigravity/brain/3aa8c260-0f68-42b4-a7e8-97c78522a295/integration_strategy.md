## 🔀 Integration Options

### Option A: **Knowledge Item (KI) 化**

**概念**: モジュールを「参照可能な知識」として KI に保存

```
artifacts/
  legacy_modules/
    first_principles.md    (← 第一原理思考)
    adversarial_review.md  (← 敵対的レビュー)
    ...
```

**Pros**:

- 即座に実装可能（ファイルコピー + metadata.json）
- 原本を保持しつつ参照可能
- 検索可能になる（Symplokē統合）

**Cons**:

- 「死蔵」になりやすい（能動的発動なし）
- 現行スキルとの重複管理が必要

---

### Option B: **Tekhne-Maker references/ 統合**

**概念**: `tekhne-maker/references/` に吸収し、プロンプト生成時の「素材ライブラリ」に

```
.agent/skills/tekhne-maker/
  references/
    cognitive_armory.md    (既存)
    legacy_modules/        (← 新規)
      first_principles.md
      ...
```

**Pros**:

- Tekhne-Maker v6.0 で自動参照
- プロンプト生成時に「先輩の知恵」として活用
- prompt-lang-generator 吸収の実績あり

**Cons**:

- Tekhne-Maker 依存になる
- 全モジュール変換コストが高い

---

### Option C: **τ-Layer Workflow 化**

**概念**: 具体的なモジュールを τ層ワークフローとして新規作成

```
.agent/workflows/
  fpp.md   ← First Principle Protocol (第一原理思考)
  vox.md   ← Vox Populi (仮想ユーザー座談会)
  ...
```

**Pros**:

- `/fpp` のように即座に発動可能
- Δ-τ 原則に沿った設計
- 既存アーキテクチャとの整合性最高

**Cons**:

- 31個全部はワークフロー過多
- 抽象Δ層との関係定義が必要

---

### Option D: **Hybrid（推奨）**

3層に分類して統合:

| 層 | 対象モジュール (目安) | 統合先 |
|:---|:---|:---|
| **Δ層強化** | 第一原理思考, 自律思考, 発散と収束 | 既存 Skill (`O1`, `O2`) の `references/` に追加 |
| **τ層ワークフロー** | 敵対的レビュー, コード監査, 仮想ユーザー | 新規 `/vox`, `/code-audit` など |
| **KI参照** | WBSスケジューリング, 形態素解析など特化系 | `knowledge/legacy_modules/` |

## 🧭 Proposed Process

1. **全モジュール棚卸し**: 31ファイルを機能分類
2. **重複チェック**: 既存スキル/ワークフローとの対応表作成
3. **吸収方針決定**: モジュールごとに A/B/C/D を判定
4. **段階的統合**: 高頻度使用モジュールから優先実装
5. **原本アーカイブ**: 統合完了後、原本を `Archive/integrated/` へ移動
