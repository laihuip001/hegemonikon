## 1. n8n vs Zapier 役割分担

| 軸 | n8n (内部処理・脳) | Zapier (外部神経・感覚) |

## 2. ワークフロー挿入ポイント

### 2.1 /boot — 認知バイアスのリセットと準備

- **n8n**: 毎朝 06:00 に Gnōsis 鮮度チェックと自動収集を実行。
- **n8n**: Perplexity Inbox から最新の調査結果を読み込み、サマリーを生成。
- **成果**: セッション開始時に「AIが昨日の世界をすでに学んでいる」状態を実現。

### 2.2 /bye — 永続化と自己反映

- **n8n**: Handoff 保存をトリガーに、Kairos/Sophia への自動インデックス投入を実行。
- **n8n**: Dispatch Log から活動統計を自動集計し、成長を可視化。

### 2.3 /sop — 外部調査プロトコル

- **Zapier**: Slack リアクション等をトリガーに Perplexity API を叩き、結果を GitHub (Mnēmē) に保存。
- **成果**: 手動のコピペ作業を排除し、AIの外部感覚器官として機能。

### 2.4 /eat — 自律消化パイプライン

- **n8n**: 収集された論文群から Digestor スコアに基づき最適な3件を自動選定。

## 5. アーキテクチャ原則 (Tier 2 Insights 2026-02-01)

洞察マイニングから抽出された自動化設計原則。

### 5.1 二層構造原則

> **「定期実行向けの二層構造：Scheduled Tasks 用の指示を汎用指示から分離する」**

自動化タスクは以下の二層に分離する：

| 層 | 内容 | 例 |
|:---|:-----|:---|
| **汎用指示** | AI の恒常的な振る舞い | Rules.md, persona.yaml |
| **定期タスク指示** | 時間/イベント駆動の特定処理 | n8n フロー, cron ジョブ |

### 5.2 公理自動チェック

> **「Python 基盤に公理の実装、System Rule に規則定義、自動発動で Creator は『当たり前』を意識せずに済む」**

自動化の最終目標は「Creatorの認知負荷ゼロ」：

1. **公理層**: FEP ベースの検証ロジック (Python)
2. **規則層**: System Rule として明文化
3. **自動発動層**: 全 WF の実行前/実行後にチェック

### 5.3 懸念停止原則

> **「『懸念が多すぎて進めない』は正しい状態」**

自動化を実装する際、懸念が多い場合は無理に進めない。これは Epochē (判断停止) の適用であり、品質を守る行動である。
