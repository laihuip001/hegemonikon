# 再試行ロジック評価者 レビュー

## 対象ファイル
`mekhane/symploke/jules_client.py`

## 発見事項
- **ジッター（Jitter）の欠如**: `with_retry` デコレータおよび `poll_session` のバックオフ計算において、ランダムなゆらぎ（Jitter）が含まれていません。バックオフ時間は決定論的に増加するため、多数のクライアントが同時に再試行を行う「サンダリング・ハード（Thundering Herd）」問題が発生するリスクがあります。
- **ポーリングループの堅牢性不足**: `poll_session` メソッドは `RateLimitError` のみを捕捉してバックオフを行っています。`get_session` が（装飾された再試行回数を超えて）`aiohttp.ClientError`（例: 502 Bad Gateway や一時的な接続断）を送出した場合、ポーリングループ全体がクラッシュします。長時間のポーリングを行うプロセスとしては、これらの一時的なネットワークエラーに対してもバックオフを行って継続すべきです。
- **ネストされた再試行による増幅**: `poll_session` が `@with_retry` で装飾された `get_session` を呼び出しているため、ポーリングの1サイクルごとに最大3回の即時再試行が行われます。これにより、障害発生時にAPIへの負荷が増幅される可能性があります。

## 重大度
- High
