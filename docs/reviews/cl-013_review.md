# エラーハンドリング一貫性評価者 レビュー

## 対象ファイル
`mekhane/symploke/jules_client.py`

## 発見事項
- **戻り値パターンの混在**: `create_session` や `poll_session` などの単体操作メソッドはエラー時に例外を送出するが、`batch_execute` などのバッチ操作メソッドは例外を捕捉して `JulesResult` オブジェクトを返す。利用者はメソッドによって例外処理と戻り値チェックを使い分ける必要があり、認知負荷が高い。
- **例外階層の漏洩**: `_request` メソッド内で `RateLimitError` はラップされているが、その他の HTTP エラーは `aiohttp.ClientResponseError` がそのまま送出されている。これにより、クライアントの利用者が実装詳細である `aiohttp` に依存することになる。
- **エラー時のデータ不整合**: `batch_execute` で例外が発生した際、`id=f"error-{uuid}"` を持つ擬似的な `JulesSession` オブジェクトが生成される。これは実在しないセッション ID であり、ログ追跡やデバッグ時に混乱を招く「Fake Object」パターンとなっている。
- **ポーリング時のエラーハンドリング非対称性**: `poll_session` は `RateLimitError` に対してのみバックオフ制御を行い、その他のネットワークエラー（5xx など）については `get_session` のリトライ装飾子に依存している。リトライ上限を超えた場合、その他のエラーは即座に例外として伝播し、ポーリングループ全体が中断されるリスクがある。

## 重大度
- Medium
