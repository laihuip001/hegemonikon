# 予測誤差バグ検出者 レビュー

## 対象ファイル
`mekhane/symploke/jules_client.py`

## 発見事項
1. **幻覚的エラー実体の生成 (Hallucinated Error Entities)**
   - `batch_execute` メソッドにおいて、例外発生時に `error_id = f"error-{uuid.uuid4().hex[:8]}"` として架空のセッションIDを生成している。
   - **FEP観点**: 外部世界（サーバー）に存在しない実体を内部モデル（クライアント）で捏造することは、エージェント間の相互予測可能性を著しく低下させる。ログ分析時にサーバーログとの突合が不可能となり、デバッグ時の自由エネルギー（驚き）を増大させる要因となる。

2. **カーディナリティによるエラー伝播の不整合 (Inconsistent Error Propagation)**
   - `create_and_poll` は例外を透過的に送出（raise）するのに対し、`batch_execute` は例外を捕捉して `JulesResult` に封じ込める。
   - **FEP観点**: 単一実行かバッチ実行かという「形式」の違いによって、エラー発生時の「振る舞い（事後分布）」が根本的に異なる。この予測困難なインターフェース設計は、利用側コードの認知負荷（モデル複雑性）を不必要に高めている。

3. **適応性のないポーリング戦略 (Rigid Polling Policy)**
   - `poll_session` は状態変化の有無にかかわらず一定間隔（5秒）でサンプリングを行う。
   - **FEP観点**: 情報獲得量（Information Gain）が低い状態（例：長時間の `PLANNING`）においてもサンプリング頻度を下げないため、代謝コスト（ネットワーク資源）の無駄が生じている。予測誤差が減少している（状態が安定している）ならば、サンプリング間隔を広げる「能動的推論」の実装が望ましい。

## 重大度
- High
