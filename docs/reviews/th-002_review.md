# 信念状態一貫性評価者 レビュー

## 対象ファイル
`mekhane/symploke/jules_client.py`

## 発見事項
- `parse_state`関数において、未知の状態（ValueError）を `SessionState.IN_PROGRESS` にマッピングしている。これは「未知の状態はアクティブである」という強い暗黙的仮定に基づいており、予期せぬ終了状態やエラー状態を隠蔽し、無限ループや誤解を招くリスクがある。
- `JulesClient`クラスにて `MAX_CONCURRENT = 60` とハードコードされており、コメントに "Ultra plan limit" とある。これは実行環境が常に「Ultraプラン」であることを暗黙的に前提としており、異なるプランでの動作時にレート制限違反を引き起こす可能性がある。
- レスポンス処理の不統一：`create_session` では `data["id"]` などを直接参照し（キーの存在を前提）、`get_session` では `.get()` を使用している。これはAPIコントラクトの信頼性に対する信念が一貫していない（作成時は完全信頼、取得時は防御的）。
- リトライロジックの非対称性：`poll_session` は `RateLimitError` に対してバックオフを行うが、`create_session` は即座にエラーを送出する。これは、ポーリング操作のみがレート制限の対象になりやすい、あるいは作成失敗は呼び出し元で処理すべきという暗黙の区分けを示唆しているが、明示されていない。
