## 概要

HGK Desktop App を、単なるダッシュボードから**認知活動の統合プラットフォーム**へ進化させる。
Obsidian + Notion + AI Web Browser のハイブリッドとして、全活動の把握・管理・自動分類・知識連携を実現する。

---

## F1: マザーブレイン (常時 Boot 機構)

> 「全活動の把握と管理に特化した認知活動の DB マネージャー」

### 要件

- ダッシュボードを**常時 Boot 機構**にする
- 新規セッションで `/boot` を不要にし、一元管理する（Boot のクラウド化）
- 全 PJ（活動）の進捗等を常に把握・更新する
- 各セッションではマザーブレインを介してデータをやり取りする
- 各セッション間の作業内容の**不整合や冗長性を防ぐ**

### Push 型インタラクション

- アプリを開いた瞬間に状況を**「語りかける（プッシュする）準備」**をしている
- 各情報項目をクリックしたら、語りかけてくれる
- 受動的なダッシュボードではなく、**能動的なナレーター**

---

## F2: セッション = ノート (動的ナレッジ)

> 「Obsidian を箱にして、中身を静的なノートから動的なセッションに入れ替える」

### 要件

- Obsidian/Notion のようにセッションをディレクトリ（PJ）毎にフォルダ分け
- セッション内容の**自動分類（ノート化）**
- 分類されたノートはセッションも兼ねる（分類後もそのままセッションの続きを行える）
- 「とりまのセッション」は Obsidian でいう**デイリーノート**
- セッション中（無理なら `/rom` や `/bye` 時に）自動で**具体を全く損ねずに**各 PJ ディレクトリに保存・分類仕訳

### MemX 的機能

- 似たテーマのメモをジャンル分け + メモ統合 → **新しいアイデアの提供**
- 過去の記事に基づいて**関連情報を取得**

### メタデータ / タグ

- 各ノート（セッション）にタグ機能（メタデータ）
- 例: 日時 / PJ 名 / 小 PJ 名

### リンク構造

- 各セッションをリンクで結びつける
- **ディレクトリ階層構造 + ネットワーク構造**の組み合わせ（FM の構造に近い）
- 階層で MECE に自動分類、ネットワーク構造で連携（情報共有・関係性・依存関係の明示）

### UI

- 左側にディレクトリ階層（開閉可能）

## F4: HGK アシスタント（サイドバー）+ AI 指揮台

> 「Comet のように、セッション外の AI を横で使える」

### 要件

- サイドバーでセッション中に、セッション外の AI エージェントで HGK を利用可能
- イメージ: Comet で Web の AI エージェントのセッションを開きながら、サイドバーでその内容を HGK に相談
- **HGK アシスタント**の実装

### 根本原則: コードを書くのは 100% AI

> 「コードエディタは不要。コードを書くのは AI（API）であり、私ではない。」 — Creator

HGK OS は **IDE (統合開発環境) ではなく、AI Orchestration Platform (AI 指揮台)**。

| 人間がやること | 必要な UI | 不要な UI |
|:-------------|:---------|:---------|
| AI の出力をレビュー | diff ビュー (変更前/後) | コードエディタ |
| 承認 / 却下 | ✅/❌ ボタン | コード補完 |
| 方向性を指示 | チャット / コマンド | シンタックスハイライト |
| 結果を確認 | ターミナル出力 (読取専用) | 書込み可能ターミナル |

必要なもの:

- **チャット UI** — AI への指示
- **Diff ビュー** — AI の出力レビュー + 承認/却下 (GitHub PR 風)
- **結果表示** — ターミナル出力 (読取専用ログビュー)
- **ファイルブラウザ** — 状態確認 (閲覧のみ、F2 ツリーと兼用)

## F5: 仮想 Twitter フィード（プッシュ型情報共有）

> 「仮想インフルエンサーが私向けの最新情報をポストする」

### 要件

- アプリ内に仮想 Twitter アカウント（システムプロンプトによる独自視点のインフルエンサー）
- 日々「私（HGK）向けの最新情報」をポスト
- 内容は**独自の記事**
  - ツイートの見出し（3行要約）
  - ツイート本文（10〜20行要約とリンク）

### いいね & 学習

- ツイートに**いいね機能** → 好みを学習
- いいねを付けたツイートは:
  - 追加の**フォローアップリサーチ**を実行
  - 常駐 Boot や他機構での**消化候補に自動追加**
  - 消化をサジェスト

### コメント機能

- ツイートに**コメント（メモやボヤキ）**を書ける
- フォローアップリサーチの参考情報に利用

## F6: AI 組織アーキテクチャ (認知コロニー)

> 「AI が AI を部下に持ち、管理する」

### 組織設計

```
Creator (CEO)
  └─ COO (Opus 4.6 = Hegemonikón の具現化)
       ├─ CTO (Claude Sonnet) — 技術判断・設計
       ├─ Engineer Lead (Gemini 3 Pro) — 高速実行
       ├─ Research Lead (Periskopē) — 調査・分析
       ├─ QA Lead (Basanos Council) — 品質監査
       └─ Intern Pool (Jules × N) — 並列コーディング
```

### マザーブレイン vs COO — 分離設計

| | マザーブレイン (F1) | COO (F4) |
|:--|:-------------|:-----------------|
| **比喩** | 巣 / DB / 海馬 | 隊長 / CPU / 前頭前皮質 |
| **性質** | 受動的 (データ保持) | 能動的 (判断・指示) |
| **状態** | 永続 (常にそこにある) | セッションに紐づく |
| **HGK** | Anamnēsis + Sympatheia | Hermēneus + Taxis + Energeia |
| **FEP** | 生成モデル (内部表現) | 能動推論 (行動選択) |

### 生物学的アナロジー

| 生物 | HGK 対応 | なぜ |
|:-----|:---------|:----|
| **アリ** | stigmergy (癒跡協調) | Mother Brain = フェロモン (= 環境を介した間接協調) |
| **ミツバチ** | ワガルダンス (民主的意思決定) | Basanos 評議会 = 偶実想による投票 |
| **人間組織** | 階層 + 専門化 | CEO → COO → 専門家チーム |
| **神経系** | 意識 + 脳 + 神経 | Creator (= 意識) + COO (= 前頭前皮質) + Workers (= 専門野) |

### HGK フレームワークの拡張利用

- **BC (行動制約)**: COO が Workers の出力に対して BC を適用 (品質管理)
- **WF (ワークフロー)**: COO がタスクを WF に分解し、Workers に委任
- **CCL**: COO が CCL 式でワークフローをディスパッチ
- **FEP**: 各 Worker の出力の予測誤差を測定し、最適な Worker にルーティング

## F7: 3D Knowledge Base (3DKB)

> 「生きたナレッジの3Dリンク + 動かせる図示」

### 統合ビジョン

既存の Three.js 96要素グラフを拡張し、**セッション・PJ・知識が同じ3D空間に共存**する。

```
3D 空間の中のノード:
  ● 96 要素 (公理/定理/関係)  ← 既存
  ◆ セッション                  ← F2 から
  ■ PJ ディレクトリ             ← マザーブレインから
  ○ 知識 (KI/Gnōsis)          ← 既存
```

### エッジ (リンク)

| リンク種別 | 意味 | 色 |
|:---------|:-----|:---|
| 定理間関係 (X-series) | 既存の72関係 | ネオンブルー |
| セッション → 定理 | どの定理がどのセッションで使われたか | オレンジ |
| セッション → PJ | セッションの所属 PJ | 緑 |
| セッション → セッション | F2 のリンク構造 | 白 |
| 知識 → セッション | いつ消化されたか | 紫 |

### 埋め込み可能な「動かせる図示」

- ノート / アーティファクト内に **インタラクティブ 3D ミニグラフ** を埋め込み
- 画像の代わりに `<3d-graph>` コンポーネントを使用
- 回転・ズーム・ノードクリックで詳細表示
- 依存関係、PJ構造、定理マップを**動的に可視化**

```markdown
例: ノート内での埋め込み

## PJ 間の依存関係
<3d-graph src="dependencies" nodes="ochema,mneme,hegemonikon" depth=2 />

## 今日の活動マップ
<3d-graph src="today" filter="sessions" />
```

### 既存資産

| 既存 | 活用 |
|:-----|:-----|
| `graph3d.ts` | Three.js + d3-force-3d の基盤 |
| `OrbitControls` | 回転・ズーム操作 |
| Graph API (`/api/graph/nodes`) | ノードデータ取得 |
| F2 リンク構造 | セッション間エッジ |

## 安全設計

> **「API で暴走は起きるのか？」 — 起きない。ただし「高速に間違い続ける」ことはありうる。**

| 操作種別 | 権限 | 安全装置 |
|:---------|:------|:--------|
| 破壊的操作 (ファイル削除, git push) | Creator 承認必須 | 既存 BC-5 + I-1 |
| 非破壊的操作 (読取, 分析) | COO 自律判断 OK | ログのみ |
| **キルスイッチ** | **BC が既にキルスイッチ** | 特別な技術不要 |
