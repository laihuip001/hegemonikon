# 比喩一貫性の詩人 レビュー

## 対象ファイル
`hermeneus/tests/test_verifier.py`

## 判定
発言（要改善）

## 発見事項
- **比喩の混在 (機械 vs 議論)**: `DebateEngine` は「議論(Debate)」のドメインに「機械(Engine)」の比喩を持ち込んでいます。これは「一つのドメインには一つの比喩世界」の原則に反します。`DebateSession` や `DebateForum`、あるいは `DebateController` など、より有機的またはプロセス指向の用語が適切です。(Medium)
- **比喩の混在 (科学/計測 vs 議論)**: `ConvergenceDetector` は「検出器(Detector)」という機械的・科学的な比喩です。議論の文脈では「合意形成」が目的であり、`ConsensusObserver`、`AgreementJudge`、`ConvergenceWatcher` などが調和します。(Medium)
- **比喩の混在 (商業/倉庫 vs 監査/記録)**: `AuditStore` は「店舗/倉庫(Store)」を連想させます。監査(Audit)の文脈では、記録保管所を意味する `AuditArchive` や `AuditRegistry` が適しています。「Store」は汎用的すぎ、抽象度の不統一も感じさせます。(Low)
- **命名の不統一 (主体)**: `DebateArgument` では `agent_role`、`RallyTurn` では `speaker` が使用されており、発言主体を示す用語が統一されていません。`DebateArgument` に合わせるなら `role` または `agent_role`、発話行為に注目するなら `speaker` ですが、ドメイン内での統一が求められます。(Low)
- **動詞の不統一 (操作)**: `AuditStore` の操作において、データを追加する `record` (動詞としての使用は一般的だが名詞と同形) と、データを取得する `get` (非常に汎用的) が対になっています。監査ログの文脈なら `log` / `retrieve` や、`append` / `read` などのペアが考えられます。またテストコード内で `retrieved = temp_store.get(...)` としており、変数名に `retrieve` を使いながらメソッドは `get` である点も揺らぎを感じさせます。(Low)

### 修正案 (提案)
1. `DebateEngine` -> `DebateSession`
2. `ConvergenceDetector` -> `ConsensusObserver`
3. `AuditStore` -> `AuditArchive`
4. `RallyTurn.speaker` -> `RallyTurn.agent_role` (または `DebateArgument.agent_role` -> `DebateArgument.speaker`)
5. `AuditStore.record()` -> `AuditStore.append()` / `AuditStore.log()`
