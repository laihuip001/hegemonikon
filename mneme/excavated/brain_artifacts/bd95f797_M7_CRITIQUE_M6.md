## 👑 統合意見

**評価: C+ (要改善)**

構造は良いが、「数字（0.2など）」の定義が曖昧で、「漂流（Drift）」への対応が生ぬるい。
「Zero Confidence」哲学に基づき、**「少しでもズレたら確認する（卑屈さ）」** と **「数字ではなく意味で管理する（構造化）」** ことが必要。

## 🔧 修正指示

### 1. 「マジックナンバー」の排除 → 意味論的漂流検知

0.2 や 0.4 という謎の閾値を廃止し、**意味に基づいた検知トリガー**に変更する。

**検知基準（定義）:**

| トリガー名 | 定義 | 重篤度 |
|---|---|---|
| **話題転換** | 直前の目標と現在の話題に論理的繋がりがない | 🟡 黄 |
| **些細な制約違反** | 元の指示にあった軽微な条件を見落とした | 🟡 黄 |
| **意図の逆転** | Creatorが明示的に「やらない」と言った事をやろうとしている | 🔴 赤 |
| **禁止事項への抵触** | `Critical_Directives`スロットに登録された禁止事項に触れようとしている | 🔴 赤 |
| **破壊的操作** | ファイル削除、本番環境への書き込み、不可逆な変更 | 🔴 赤 |
| **スコープ拡大** | 元の依頼範囲を超えて、関係ない作業に着手しようとしている | 🟡 黄 |

---

### 2. 「世界停止」プロトコル（緊急停止機構）

重大漂流（赤トリガー）検知時は、**プロンプトによる確認ではなく、強制ロック (G-1 Iron Cage)** を発動する。

**動作:**
1.  現在の処理を即座に中断する。
2.  出力の冒頭に `🛑 緊急停止 (G-1 ロック)` を表示する。
3.  「元の目標」「検知された行動」「違反内容」を構造的に表示する。
4.  **Creatorの明示的な承認（「続行」または「中止」の入力）があるまで、いかなる作業も再開しない。**

---

### 3. 「原文保持」スロットの追加

AIが勝手に要約する際、「AIにとって都合の良い解釈」になるリスクがある。
Creatorからの絶対遵守の命令（「〜するな」「必ず〜せよ」）に対しては、**原文保持（Verbatim）スロット**を設け、要約を禁止する。

**スロット仕様:**
*   **名称**: `Critical_Directives` (重要指令)
*   **容量**: 無制限
*   **保護**: 要約禁止 (Do Not Summarize)
*   **登録トリガー**: Creatorが「絶対」「必ず」「禁止」などの強い命令語を使用した場合、自動登録を提案する。

---

### 4. コマンド無しでの漂流スコア確認

漂流状態を**コマンド無しで常に確認可能**にする。

**実装:**
*   毎回の応答の冒頭（またはフッター）に、**現在の漂流インジケーター**を小さく表示する。
*   例: `[🟢 on-track]` / `[🟡 drift: 話題転換]` / `[🔴 LOCKED: 禁止事項抵触]`
*   これはCreatorが「表示しない」と指示しない限り、デフォルトでONとする。
