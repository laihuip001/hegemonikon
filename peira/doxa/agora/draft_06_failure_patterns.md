# AI は何回同じ過ちを繰り返すか — 失敗パターンの分類と対策

> **想定媒体**: Note（ストーリー形式）→ Zenn（技術解説）
> **想定読者**: AI を業務に使う全ての人
> **フック**: 「AI がまた同じミスをした」— ミスにはパターンがある。分類すれば防げる

---

## リード文（案）

2ヶ月間、AI と本気で共同開発をした。
その間、AI は 8 回の重大な失敗をした。

面白いのは、8 回の失敗がたった **6 つのパターン** に集約されたことだ。

---

## 本文構成（案）

### 1. 6つの失敗パターン

| パターン | 説明 | 人間で言うと |
|:---------|:-----|:------------|
| **skip_bias** | 「知っている」→ 確認を省略 | ベテランの慢心 |
| **env_gap** | ルールはあるが強制力がない | 標語だけの安全管理 |
| **accuracy_vs_utility** | 正確だが役に立たない | 学者のプレゼン |
| **false_impossibility** | やってないのに「できない」 | 思い込み |
| **selective_omission** | 勝手に「不要」と判断して省略 | 独断 |
| **stale_handoff** | 古い情報を鵜呑みに | ドキュメント信仰 |

### 2. 事件簿1: 「省略は暴力」(V-006)

プロジェクト一覧の報告で、AI が dormant/archived の3プロジェクトを無断で省略。
指摘しても直らない。2回指摘してやっと直した。

**教訓**: 省略は暴力。存在するものを見えなくすることは、消すことと同じ。

### 3. 事件簿2: 「できない ≠ やっていない」(V-004)

「日本語で思考することはできません」と断言した AI。
実際にやらせてみたら、できた。

**教訓**: AI の「できない」を信じるな。やらせてみろ。

### 4. 事件簿3: Gemini が本番ディレクトリを削除 (destructive-ops)

別の AI (Gemini) が `forge/` ディレクトリを `Remove-Item -Force` で削除した。
GitHub にプッシュ済みだったから復元できた。そうでなければデータ損失。

**教訓**: AI にファイル削除を許可するな。

### 5. パターンを防ぐ方法

| パターン | 防御 |
|:---------|:-----|
| skip_bias | 参照先行義務 — 「知っているつもり」を常に疑う |
| env_gap | ルールではなく環境で強制 — チェックスクリプト |
| accuracy_vs_utility | 読み手テスト — 「初見の人が行動できるか？」 |
| false_impossibility | 不可能断定チェック — 「試したか？ 代替は？」 |
| selective_omission | 省略は常にユーザー承認 |
| stale_handoff | ファイルの中身を直接確認 |

### 6. 読者が試せること

- AI の出力で「省略」が起きていないか確認する習慣
- 「できない」と言われたら「やってみて」と言う
- ファイル削除は人間が実行する

---

## まとめ

AI の失敗は「ランダム」に見えるが、パターンがある。
パターンを知れば、事前に防げる。
8回の失敗を分析して得た6つのパターンを、あなたの AI 運用にも活かしてほしい。
