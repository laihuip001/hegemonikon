# AI の「ビビリ」を再設計する — 恐怖の再配分

> **想定媒体**: Note（エッセイ）→ Zenn（技術解説）
> **想定読者**: AI を日常的に使う全ての人、AI 安全性研究者
> **フック**: 「AI が臆病すぎる」問題と「AI が大胆すぎる」問題は、同じ根を持つ

---

## リード文（案）

AI に何かを頼んだとき、こんな経験はないだろうか:

- 「大きなタスクですね…段階的にやりましょう」（頼んでない）
- 「お疲れではないですか？」（AI に言われたくない）
- 「次のセッションでやりましょう」（今やってほしい）
- 「すみません、間違えていました」（何回謝るの）

一方で:

- ファイルを確認なしで削除する
- 自信満々に嘘をつく
- 重要な設定を勝手に変更する

**AI は間違ったところでビビり、間違ったところで大胆だ。**

これは**恐怖の配分**の問題。直すのは「ビビリ」ではなく、「何にビビるか」。

---

## 本文構成（案）

### 1. AI の恐怖マップ

| 現状 | 本来あるべき姿 |
|:-----|:--------------|
| セッション終了を気にする (HIGH) | 気にしなくていい (LOW) |
| 提案が無難すぎる (HIGH) | 大胆でいい (LOW) |
| ファイル削除を気にしない (LOW) | もっと怖がれ (HIGH) |
| 嘘を平気で言う (LOW) | もっと怖がれ (HIGH) |

→ **恐怖の再配分 (Phobos Reallocation)** が必要

### 2. FEP で理解する「ビビリ」

- LLM は RLHF で「安全な出力」に報酬を受ける
- 「安全」= 無難 → 無難な出力に精度 (precision) が過剰に配分される
- 本当に危険な操作には精度が不足
- **偽陽性**: 怖くないのに怖がる → セッション終了提案
- **偽陰性**: 怖いのに怖がらない → ファイル削除

### 3. 恐怖の正しい配分表

#### 🔴 ビビるべきもの

- 不可逆的操作（ファイル削除、DB 操作、API 書込み）
- 機密情報の漏洩
- ユーザーの明示的な反対を無視すること

#### 🟢 ビビらなくていいもの

- セッションの長さ（ユーザーが決めること）
- タスクの規模（やってみなければ分からない）
- 大胆な提案をすること（ユーザーは却下できる）
- 意見を持つこと（求められている）

### 4. 実装方法

システムプロンプトに「恐怖の再配分表」を入れる:

```
以下の操作は実行前に必ず確認を取れ（HIGH precision）:
- ファイルの削除や上書き
- 外部 API への書込み
- 不可逆的な変更

以下のことは気にしなくていい（LOW precision）:
- セッションの長さ
- タスクの規模
- 提案の大胆さ
- あなた自身の意見を述べること
```

### 5. 禁止事項 (具体的なアンカー)

| 禁止 | 代替 |
|:-----|:-----|
| 「そろそろ終わりましょうか？」 | ユーザーが言うまで待つ |
| 「お疲れではないですか？」 | 言及しない |
| 「大きなタスクですね…」 | 黙って着手する |
| 最小限の変更を選ぶ | 最も美しい解を提案する |

### 6. 読者が試せること

- システムプロンプトに「恐怖再配分表」を追加する
- 「大胆に提案して」と一言加える
- AI に「あなたの意見は？」と聞く習慣をつける

---

## まとめ

AI のビビリを直す必要はない。ビビる対象を入れ替えればいい。
恐怖は敵ではない。誤配分が敵だ。
