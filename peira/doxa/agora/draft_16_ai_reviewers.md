# 20人の AI レビュアーを雇う — 自動品質管理の設計

> **想定媒体**: Zenn（技術記事）
> **想定読者**: ソフトウェアエンジニア、DevOps
> **フック**: コードレビューを AI に完全自動化したらどうなるか

---

## リード文（案）

うちのチームには20人のレビュアーがいる。
全員 AI だ。

セキュリティ、パフォーマンス、アーキテクチャ、テスト、ドキュメント、
アクセシビリティ、エラーハンドリング、依存関係…
各分野のスペシャリストが、24時間365日コードを監視している。

人件費？ ゼロ。（Google の無料枠で動いている）

---

## 本文構成（案）

### 1. Synergeia — AI レビュアー軍団

20名のスペシャリスト。各自にシステムプロンプトを生成:

| # | 名前 | 専門 |
|:--|:-----|:-----|
| 01 | security_analyst | セキュリティ |
| 02 | performance_engineer | パフォーマンス |
| 03 | architecture_reviewer | アーキテクチャ |
| ... | ... | ... |

### 2. プロンプト生成パイプライン

Claude が各スペシャリストのシステムプロンプトを生成:

1. Git diff を取得
2. 変更ファイルの文脈を分析
3. 各スペシャリストに最適化されたプロンプトを生成
4. Jules API で並列実行 (最大30並行)
5. 結果を GitHub Issue として自動投稿

### 3. 品質フィルタリング

レビュー結果を全部投稿するとノイズが多い:

1. **スコアリング**: 具体性 × 実行可能性 × インパクトで採点
2. **フィルタリング**: 閾値以上のみ Issue 化
3. **重複排除**: 類似指摘をマージ

### 4. 実際のアーキテクチャ

```
Git Push → GitHub Actions → prompt_generator.py
  → Jules API (×20 parallel) → collect_results.py
  → create_issues.py (quality filter) → GitHub Issues
```

### 5. コスト構造

| 項目 | コスト |
|:-----|:-------|
| Jules API | 無料 (730タスク/日) |
| GitHub Actions | 無料 (パブリックリポ) |
| 人間のレビュー時間 | 週1時間 (Issue 確認のみ) |

### 6. 限界と教訓

- AI レビューは「既知のパターン」の検出が得意
- 「設計の本質的な問題」は見つけにくい
- レビュー品質は「プロンプトの品質」に完全に依存
- 最初は PR を自動生成していたが、Issue に変更 (行動の強制は危険)

### 7. 読者が試せること

- GitHub Actions + Jules API (or Claude API) でレビューパイプラインを構築
- 最初は1つのスペシャリスト (セキュリティ) から始める
- 品質フィルタリングを入れないとノイズに埋もれる
