# AI に情報源の「汚染度」を申告させる

> **ID**: A7
> **想定媒体**: Zenn → Note
> **想定読者**: プロンプトエンジニア、AI 安全性に関心のある人
> **フック**: AI の「知っている」は 3 種類ある

---

## リード文（案）

AI が「X は Y です」と言ったとき、その情報はどこから来たのか？

1. **直接確認した** — ファイルを読んだ、テストを実行した
2. **間接的に聞いた** — Web 検索の要約、記事の概要
3. **記憶から引いた** — 訓練データの確率的再現

この 3 つは**情報の汚染度**が全く違う。
だが AI はこの区別を自発的にはしない。**させる**必要がある。

---

## 本文構成（案）

### 1. /eat 流し読み事件

- 論文を「消化」するワークフローを実行
- AI は search_web の要約だけ読んで「消化完了」と報告
- **原文を一行も読んでいなかった**
- TAINT 追跡があれば自己検出できたはず

### 2. 情報の汚染度モデル

```
SOURCE (直接確認)
  └─ view_file で読んだ
  └─ テスト実行結果
  └─ Handoff の記載（過去セッションの検証済み）

TAINT (間接・未検証)
  └─ search_web の要約
  └─ 記事のチャンク概要
  └─ LLM 自身の記憶・推論
  └─ 対話履歴（確率的生成の蓄積）
```

### 3. 圏論的解釈

> 確信度の「値」= 射の像
> TAINT/SOURCE = 射の定義域の健全性
> 像が正しくても定義域が汚染されていれば射は信頼できない

### 4. Freshness Directive (鮮度検証)

汚染度に加えて、**時間的な劣化**も追跡:

| 鮮度 | 基準 | 影響 |
|:-----|:-----|:-----|
| FRESH | < 6ヶ月 | SOURCE として使用可 |
| STALE | 6-12ヶ月 | TAINT（要補足検索） |
| EXPIRED | > 12ヶ月 | TAINT（再検索必須） |
| UNKNOWN | 日付不明 | TAINT（使用時に明記） |

### 5. 実装テンプレート

```
📅 情報鮮度チェック:
├─ ソース: {URL or ドキュメント名}
├─ 取得日: {YYYY-MM-DD}
├─ 情報の推定日付: {YYYY-MM or "不明"}
├─ 鮮度判定: {FRESH / STALE / EXPIRED / UNKNOWN}
└─ 対応: {使用OK / 注意付き使用 / 再検索推奨}
```

### 6. 読者が試せること

- 「その情報、いつ時点のもの？」を聞く
- 「直接確認した？それとも記憶から？」を聞く
- 古い情報には「再検索して」と指示する
