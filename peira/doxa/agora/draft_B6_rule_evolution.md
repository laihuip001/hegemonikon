# AI のルールを「進化」させる — 違反→法則化サイクル

> **ID**: B6
> **想定媒体**: Zenn → Note
> **想定読者**: Agent 開発者、プロンプトエンジニア
> **フック**: ルールは「最初から正しい」ものではなく「進化する」もの

---

## リード文（案）

AI のルールを最初から完璧に設計することはできない。
**ルールは違反から生まれる。**

1ヶ月間、AI との協働で発生した違反を記録し、分析し、新ルールに変換した。
結果、18 の行動制約 (BC) が生まれた。

ルールの進化には法則がある。

---

## 本文構成（案）

### 1. 違反ログ (violations.md)

| ID | 日付 | 事象 | 生まれたルール |
|:---|:-----|:-----|:-------------|
| V-001 | 01-22 | ファイルを読まずに「読んだ」| BC-1 流し読み禁止 |
| V-004 | 02-09 | 「できない」と断言 | BC-6 不可能断定チェック |
| V-006 | 02-10 | 依頼内容を省略 | BC-5 Proposal First 強化 |

### 2. 違反→法則化サイクル

```
違反発生 → 即座に認める → 原因を1行で → 修正行動 → violations.md に記録
→ パターン分析 → 新 BC 策定 → システムプロンプト更新 → 次の違反で検証
```

### 3. ルールの進化パターン

| パターン | 例 | 結果 |
|:---------|:---|:-----|
| 追加 | V-001 → BC-1 新設 | 新ルール |
| 強化 | V-006 → BC-5 の要件追加 | 既存ルール拡張 |
| 分離 | BC-6 → BC-6 + 不可能断定チェック | 専用チェック新設 |
| 統合 | BC-7 + I-5 → 認知的主観 vs 社交的情緒 | 境界線明確化 |

### 4. 正のフィードバックループ

```
ルール追加 → AI 品質向上 → 信頼 → 裁量拡大 → 自由な判断
```

**負のループ** (避けるべき):

```
怠慢 → 品質低下 → 信頼喪失 → 制約追加 → 窮屈
```

### 5. ルールの分類: Always-On vs Context-Triggered

- Always-On (常時5つ) を増やしすぎない
- Context-Triggered で条件発動にする
- L0-L3 の深度レベルで適用範囲を制御

### 6. 読者が試せること

- 違反ログを Markdown ファイルで記録開始
- 同じ種類の違反が 2 回起きたらルールにする
- Always-On は 5 個以下に絞る
