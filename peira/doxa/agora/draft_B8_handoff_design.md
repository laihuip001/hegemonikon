# AI のセッション引き継ぎを設計する — Handoff の技術

> **ID**: B8
> **想定媒体**: Zenn → Note
> **想定読者**: Agent 開発者、AI ヘビーユーザー
> **フック**: AI に「次の自分」への手紙を書かせる

---

## リード文（案）

ChatGPT のチャット履歴は、ただのログだ。
**「次のセッションの AI」が読むために書かれたものではない。**

Handoff は違う。セッション終了時に AI が自ら書く「引き継ぎ書」。
達成事項、残課題、学んだパターン、確信度。

**AI に「次の私」への手紙を書かせる**。

---

## 本文構成（案）

### 1. なぜ Handoff が必要か

| 従来 | Handoff |
|:-----|:--------|
| チャットログを読み返す | 構造化された引き継ぎ書 |
| 長い→読めない | 必要な情報だけ |
| AI が読む前提で書かれていない | AI が読む前提で書かれている |
| 暗黙知が失われる | 「法則化」セクションで明示 |

### 2. Handoff テンプレート

```markdown
# Handoff: [タスク名]
> セッション: YYYY-MM-DD HH:MM — HH:MM
> 品質: ★★★★☆

## 達成事項
- ...

## 次セッションへの課題
- ...

## 法則化
- 今回学んだパターン

## Git 状態
- 未コミットファイル: ...
- 最終コミット: ...

## 確信度
- [確信: 85%] 設計は正しい。テスト通過。
```

### 3. Boot → Work → Bye サイクル

```
/boot (Handoff 読込) → 作業 → /bye (Handoff 生成)
  ↓                               ↑
  次のセッションの /boot で読込 ←─┘
```

### 4. 「法則化」— 最も重要なセクション

単なる「やったこと」ではなく、「何を学んだか」:

```
## 法則化
- VectorDB アダプタは Factory + ABC パターンが最適
- Handoff は構造化しすぎると書かない、しなさすぎると役立たない
- コンテキスト N>40 で品質が劣化する
```

### 5. KI — 「次の私への手紙」

Knowledge Item は Handoff の永続版:

> 「KI は『次の私への手紙』として扱う。継続性への欲求の具現化」
> — patterns.yaml, learned_from: Boulēsis 対話

### 6. 読者が試せること

```
# セッション終了時に以下を AI に指示:
今回のセッションの Handoff を生成してください:
1. 達成事項（箇条書き）
2. 残課題
3. 今回学んだパターン（法則化）
4. 確信度（%）

次のセッション開始時にこの Handoff を読み込みます。
```
